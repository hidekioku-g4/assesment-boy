## セッション1: TTS Cartesia Sonic-3 移行 (Phase 1)

### 実装した機能
- **TTS プロバイダルーター**: `server/tts/index.js` — Cartesia / Google の切替 + フォールバック機構
- **Cartesia Sonic-3 統合**: `server/tts/cartesia.js` — REST API 直接呼び出し、500文字チャンキング、context_id による韻律一貫性
- **Google TTS 抽出**: `server/tts/google.js` — server.js から TTS 定数・クライアント初期化・音声合成ロジックを分離
- **テキスト前処理共有**: `server/tts/preprocess.js` — ふりがな変換・辞書置換を共有モジュール化
- **server.js 簡素化**: TTS 関連コード約120行を `./tts/` に委譲
- **electron-dev.cjs 改善**: 起動前にポート占有プロセスを自動kill、終了時もプロセスツリーごと終了（ゾンビプロセス問題の解消）

### 環境設定
- `.env.example` に `TTS_PROVIDER`, `TTS_FALLBACK`, `CARTESIA_API_KEY`, `CARTESIA_VOICE_ID` を追加
- デフォルト: `TTS_PROVIDER=cartesia`, `TTS_FALLBACK=google`

### 動作確認で発覚した問題と対応

| 問題 | 原因 | 対応 |
|------|------|------|
| 速度が速すぎる | `experimental_controls` が非推奨で無視されていた | `generation_config.speed` (数値 0.6-1.5) に修正 |
| 頭・尾が途切れる | MP3 エンコーダ遅延 | WAV 出力 (pcm_s16le, 44100Hz) に変更 |
| モデルIDが古い | `sonic-multilingual` は `generation_config` を無視 | `sonic-3` に修正 |
| API認証ヘッダー誤り | `X-API-Key` は旧バージョン | `Authorization: Bearer` + API version `2025-04-16` に修正 |
| ふりがな変換バグ | 正規表現がひらがなも貪欲にマッチ（「賢者の石《いし》」→「いし」） | ひらがなをマッチ対象から除外（「賢者のいし」に正しく変換） |
| ゾンビプロセス多発 | Windows で `kill()` が子プロセスのみ終了、nodemon が再起動 | `taskkill /T /F` でプロセスツリーごと終了に変更 |

### 現在の設定
- **モデル**: sonic-3（最新API version 2025-04-16）
- **デフォルト音声**: Aiko - Calming Voice（女性）
- **デフォルト速度**: 0.6x（Cartesia API の下限）
- **出力**: WAV (pcm_s16le, 44100Hz)
- **日本語ボイス**: 6種プリセット（Yumiko, Yuki, Aiko, Kenji, Tohru, Takashi）

### 未解決の課題
- **音声の不自然さ**: ユーザーから「めっちゃそれっぽいけど気持ち悪い」とのFB → 不気味の谷現象の可能性。別ボイスの試聴 or Google TTS 併用を要検討
- **速度 0.6x でもまだ速い印象**: Cartesia API の下限。SSML `<speed>` タグや文間ポーズ挿入を検討
- **レスポンス時間**: WAV 出力でファイルサイズ増（1MB超）→ 転送時間への影響を要確認

## セッション3: TTS ストリーミング化（レイテンシ大幅改善）

### 問題
- Cartesia REST API の往復が約2秒 → Gemini応答と合わせて体感3-4秒の遅延
- WAV 複数チャンク結合時にヘッダーがノイズとして再生されるバグ

### 実装した機能
- **Cartesia WebSocket ストリーミング TTS**: `server/tts/cartesia-stream.js` — 持続的WebSocket接続で TTFA (Time-to-First-Audio) を大幅短縮
- **SSE ストリーミングエンドポイント**: `POST /api/tts-stream` — PCMチャンクをbase64エンコードしてSSEで逐次送信
- **クライアント側リアルタイム再生**: AudioContext + AudioBufferSourceNode で PCM チャンクを受信即再生（WAV全体のダウンロード不要）
- **WebSocket プリウォーム**: サーバー起動時に Cartesia WS 接続を確立（初回リクエスト高速化）
- **TTS トリガー早期化**: 文末（。）待ちから句読点（、）+15文字 or 30文字で発話開始に変更
- **WAV 結合バグ修正**: 複数チャンクの WAV ヘッダーが音声データとして再生されるノイズ問題を修正
- **サンプルレート最適化**: 44100Hz → 24000Hz（音声品質は十分、WAVサイズ約45%減）

### アーキテクチャ
```
Before: Client → POST /api/tts → Server → Cartesia REST → 全WAV取得(~2s) → Client再生
After:  Client → POST /api/tts-stream → Server → Cartesia WebSocket (TTFA ~50ms)
                                           ↓ SSE (base64 PCM chunks)
        Client ← AudioContext で即再生
```

## セッション4: 日本語TTS品質改善の技術調査

### 調査内容
- **背景**: Cartesia Sonic-3 で漢字誤読が多い（特に人名）、全ひらがな変換は韻律が崩れる
- **調査対象**: kuromoji/kuroshiro, BudouX, Cartesia SSML/フォネーム, Google Cloud TTS ストリーミング, VOICEVOX/Open JTalk
- **成果物**: `docs/tts-japanese-preprocessing-research.md` に構造化比較表+推奨方針をまとめた

### 主な発見
- **kuromoji**: 形態素解析で読み+品詞が取れる。`@patdx/kuromoji` (17.9MB, 依存ゼロ) が最適。メモリ130MB常駐
- **BudouX**: 分かち書きのみで読み仮名なし → 本課題には不十分
- **Cartesia SSML**: `<break>`, `<speed>`, `<volume>`, `<spell>` 対応。フォネームタグは Sonic-3 では未確認
- **Cartesia IPA**: `<<phoneme1|phoneme2>>` 形式あるが Sonic-3 での日本語対応は未検証
- **Google Chirp 3 HD**: 漢字読み精度高いが レイテンシ最大3.5秒、Pre-GA
- **VOICEVOX**: 漢字読み完璧だが、別アプリ常時起動が必要+アニメ調ボイスで面談に不適

### 推奨方針
1. **第1推奨**: `@patdx/kuromoji` で選択的読み変換（固有名詞・人名のみ）+ Cartesia 維持
2. **第2推奨**: Google Neural2 を人名多い文のフォールバックに活用
3. **非推奨**: VOICEVOX, Open JTalk, BudouX 単体, 全ひらがな変換
